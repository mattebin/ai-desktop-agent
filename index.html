AI Desktop Operator

A local AI desktop operator that can understand your system, take actions, and work through tasks step by step with controlled execution.

It combines chat, approvals, desktop evidence, Gmail workflows, automations, replay, and short-horizon memory into one operator workflow that is meant to be useful in real work, not just interesting in a demo.

Open repository See the interface

What it is

  • Not a toy agent It is built around approvals, replay, recovery, and bounded control instead of free-running automation.
  • Local by default The main control surface is a local API with a thin desktop app on top.
  • Designed to adapt It can evaluate whether actions worked, recover carefully, and remember what helped recently.

Key capabilities

High-level product capabilities, without forcing readers straight into implementation details.

  • Desktop interaction Inspect windows, browser state, and desktop evidence, then act through bounded controls.
  • Gmail workflows Review inbox threads, prepare replies or forwards, and keep sending approval-gated.
  • Automations Schedule recurring work, track watch conditions, and review triggered tasks.
  • Controlled shell lab Experiment with sandbox-first shell access in a separate lab lane.
  • Approval + safety Sensitive actions are reviewed instead of silently executed.
  • Run history and replay Inspect what happened later instead of trusting a black box.
  • Memory and learning Use short-horizon memory to reuse what worked and avoid repeating bad approaches.
  • Workflows and extensions Skills, commands, and local extensions are part of the same operator surface.

What makes it different

  • Approval-gated and bounded It is meant to stay useful without becoming reckless.
  • Evidence-based reasoning Desktop state, scene context, and tool results help decide what to do next.
  • Recovery-aware It can retry carefully, recover, and stop when progress is unclear.

How it works

The simple version is:

observe -> plan -> act -> evaluate -> adapt

The operator gathers context, chooses a bounded step, executes it, checks what actually changed, and then either continues, recovers, asks for approval, or stops.

UI overview

A chat-first interface with workspaces, approvals, and context kept visible without turning the app into a dashboard.

Chat in the center Workspaces on the left Approvals and context on the right

The center stays focused on the task. Workspaces such as Gmail, Automations, Workflows, Runs, and the experimental lab are available without crowding the main conversation.

Example use cases

  • Open a file and send it by email Inspect the result, draft the email, and wait for approval before sending.
  • Inspect something on the desktop and act Recover focus, judge readiness, and continue based on what is actually visible.
  • Run an approval-aware task Keep the operator moving while still pausing at the right moments.
  • Experiment in lab mode Try safe shell tasks without promoting that capability into the normal operator path.
Current status: the project has a serious working foundation and a real product shape, but it is still evolving. The main operator flow is usable today, while experimental areas such as shell lab mode are intentionally kept separate until they prove themselves further.
